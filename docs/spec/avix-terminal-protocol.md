## 1. Overview

ATP is the **single external surface** through which all clients interact with Avix. It replaces ad-hoc REST endpoints with a structured, authenticated, bidirectional channel.

```
┌──────────────────────────────────────────────────────────────┐
│  External Clients                                            │
│  ┌──────────┐  ┌──────────┐  ┌─────────────┐  ┌─────────┐ │
│  │ GUI SPA  │  │  CLI     │  │  Desktop App│  │  CI/CD  │ │
│  └────┬─────┘  └────┬─────┘  └──────┬──────┘  └────┬────┘ │
└───────┼─────────────┼────────────────┼───────────────┼──────┘
        └─────────────┴────────────────┴───────────────┘
                         WSS (TLS 1.3)
                    wss://localhost:7700/atp
                              │
                    ┌─────────▼──────────┐
                    │    gateway.svc     │  ← ATP server
                    │  (ATP → IPC)       │
                    └─────────┬──────────┘
                              │ IPC (local sockets)
                    ┌─────────▼──────────┐
                    │  Kernel + Services │
                    └────────────────────┘
```

### Design Principles

- **One protocol, all operations.** Every user action is a message on this channel.
- **Token per message.** The ATPToken is re-validated on every incoming command — not just on connect. Revocation is immediate.
- **Server push is first-class.** The kernel pushes events without polling.
- **Deny by default.** Unauthenticated or under-privileged messages are rejected before touching kernel state.
- **Admin port separation.** Destructive system operations only on port 7701.

## 2. Transport Layer

### Endpoints

|Port  |Path  |Binding                                        |Purpose                                |
|------|------|-----------------------------------------------|---------------------------------------|
|`7700`|`/atp`|`localhost` (desktop/CLI) or `0.0.0.0` (server)|User operations                        |
|`7701`|`/atp`|`localhost` always                             |Admin operations (`sys`, `cap` domains)|

### TLS

- TLS 1.3 minimum (TLS 1.2 allowed in dev mode via `boot.conf: tls.min_version`)
- Desktop/CLI: self-signed cert auto-generated by `gateway.svc` at first boot
- Server/Docker: operator-supplied cert via `boot.conf: tls.cert_path / tls.key_path`

### Keep-alive

Server sends WebSocket `PING` every 30 seconds. No `PONG` within 10 seconds → connection closed, `SessionManifest` set to `state: closed`.

## 3. Authentication and Session Tokens

### 3.1 Token Structure (ATPToken)

Signed YAML blob, base64url-encoded. Structurally similar to JWT but Avix-native.

```yaml
header:
  alg: HS256
  typ: ATP
payload:
  sub: alice
  uid: 1001
  role: operator               # guest | user | operator | admin
  crews: [researchers, writers]
  sessionId: sess-alice-001
  iat: 2026-03-20T10:00:00Z
  exp: 2026-03-20T18:00:00Z
  scope: [proc, signal, fs, snap, cron]
signature: sha256:hmac...      # HMAC-SHA256, keyed by kernel secret
```

Clients treat the token as an opaque string. Modification invalidates the signature.

### 3.2 Role Hierarchy and Domain Access

```
admin    ← full system control
  operator  ← spawn/kill, install services, manage crontab, view all
    user     ← own agents/workspace
      guest  ← read-only
```

|Domain  |guest|user    |operator   |admin|
|--------|:---:|:------:|:---------:|:---:|
|`proc`  |r    |rw (own)|rw (all)   |rw   |
|`signal`|—    |own     |all        |all  |
|`fs`    |r    |own     |all        |all  |
|`snap`  |—    |own     |all        |all  |
|`cron`  |—    |own     |all        |all  |
|`users` |—    |self    |—          |rw   |
|`crews` |r    |r       |r          |rw   |
|`cap`   |—    |—       |inspect    |rw   |
|`sys`   |—    |—       |status/logs|rw   |
|`events`|r    |own     |all        |all  |

**Admin port rule:** `sys` (mutating ops) and `cap` only on port `7701`.

### 3.3 Obtaining a Token

```http
POST /atp/auth/login
Content-Type: application/json

{ "identity": "alice", "credential": "sk-avix-..." }
```

Response:

```json
{ "token": "<ATPToken>", "expiresAt": "...", "sessionId": "sess-alice-001" }
```

Then upgrade:

```http
GET /atp HTTP/1.1
Upgrade: websocket
Authorization: Bearer <token>
```

### 3.4 Token Refresh

Server pushes `token.expiring` event when < 5 minutes remain. Client sends:

```json
{ "type": "cmd", "id": "r-001", "token": "<current>", "domain": "auth", "op": "refresh", "body": {} }
```

### 3.5 Token Invalidation

Tokens are invalidated on: explicit `auth.logout`, admin `cap.revoke`, user deletion, or kernel restart (all tokens are ephemeral — never persisted).

## 4. Connection Lifecycle

```
Client                              gateway.svc
  │  POST /atp/auth/login               │
  │ ────────────────────────────────►   │  auth.svc validates credential
  │  200 { token, sessionId }           │
  │ ◄────────────────────────────────   │
  │                                     │
  │  GET /atp  Authorization: Bearer    │
  │ ────────────────────────────────►   │  create SessionManifest
  │  101 Switching Protocols            │
  │ ◄────────────────────────────────   │
  │                                     │
  │  { event: "session.ready" }         │
  │ ◄────────────────────────────────   │  pushed immediately
  │                                     │
  │  { cmd, domain, op, body }          │
  │ ────────────────────────────────►   │  re-validate token
  │  { reply }                          │
  │ ◄────────────────────────────────   │
  │                                     │
  │  { event: "agent.output" }          │  pushed anytime
  │ ◄────────────────────────────────   │
```

**Reconnect grace window:** 60 seconds. Session held in `state: idle`. Running agents not stopped. After window, session closed and agents receive `SIGSTOP`.

## 5. Message Framing

All messages are JSON text frames.

### 5.1 Client → Server: Command

```json
{
  "type": "cmd",
  "id": "c-0042",
  "token": "<ATPToken>",
  "domain": "proc",
  "op": "spawn",
  "body": { ... }
}
```

Token re-validated on every message. HMAC check is < 1µs — no overhead concern.

### 5.2 Server → Client: Reply

```json
{ "type": "reply", "id": "c-0042", "ok": true, "body": { ... } }
```

Error:

```json
{ "type": "reply", "id": "c-0042", "ok": false, "error": {
    "code": "EPERM", "message": "Role 'user' may not invoke domain 'sys'",
    "detail": { ... } } }
```

### 5.3 Server → Client: Event (Push)

```json
{ "type": "event", "event": "agent.output", "sessionId": "sess-alice-001",
  "ts": "2026-03-20T10:12:34.123Z", "body": { ... } }
```

Events are scoped by role — clients only receive events they are permitted to see.

### 5.4 Client → Server: Subscription

```json
{ "type": "subscribe", "id": "sub-001", "token": "<ATPToken>",
  "events": ["agent.output", "agent.status", "tool.changed"] }
```

Use `["*"]` for all permitted events.

## 7.1 Human-in-the-Loop (HIL) — Full Specification

HIL covers three distinct scenarios, each producing a `hil.request` event with a
different `type` and requiring a different response:

|Type                |Trigger                                                                       |Token effect                                       |
|--------------------|------------------------------------------------------------------------------|---------------------------------------------------|
|`tool_call_approval`|Agent has the tool; policy requires human sign-off on this specific invocation|None — tool already granted                        |
|`capability_upgrade`|Agent requests a tool NOT in its current CapabilityToken                      |Kernel issues new token with tool added if approved|
|`escalation`        |Agent sends `SIGESCALATE` — proactively asks for guidance                     |None — guidance injected into LLM context          |

### The ApprovalToken

Every `hil.request` carries a single-use `approvalToken`. The kernel:

- Mints one per HIL event, bound to its specific context (pid + type + tool where applicable)
- Stores it in memory only — never on disk
- Atomically consumes it on first valid `SIGRESUME` response
- Any subsequent use of the same token returns `EUSED`
- Expires after `kernel.yaml: safety.hil_timeout` (default: 10 minutes) — kernel auto-denies on expiry

**Multiple open clients** (two browser tabs, GUI + CLI) both receive the `hil.request` event. The first to respond wins. The second gets `EUSED`.

The pending HIL request is also readable via `fs.read` on `/proc/<pid>/hil-queue/<hil-id>.yaml` so a client that missed the push event can catch up on reconnect.

-----

### Scenario 1 — Tool Call Approval

The agent already has the tool in its CapabilityToken but the safety policy (`hilOnEscalation: true` or a `blockedToolChains` pattern) requires human confirmation before execution.

**Flow:**

```
RuntimeExecutor intercepts tool call: send_email
  ├── CapabilityToken check: send_email IS granted ✓
  ├── Policy check: send_email requires hil approval
  │
  └── Sends ResourceRequest to kernel:
        resource: tool_call_approval
        tool: send_email
        args: { to: "team@org.com", subject: "Summary ready" }

Kernel:
  ├── Mints ApprovalToken bound to (pid:57, tool:send_email, args_hash)
  ├── Writes /proc/57/hil-queue/hil-001.yaml
  ├── Sends SIGPAUSE to agent — agent suspends at tool boundary
  └── Pushes hil.request to subscribed ATP clients
```

**`hil.request` event body:**

```json
{
  "hilId": "hil-001",
  "pid": 57,
  "agentName": "researcher",
  "type": "tool_call_approval",
  "tool": "send_email",
  "args": { "to": "team@org.com", "subject": "Summary ready" },
  "prompt": "Agent wants to send an email. Approve?",
  "approvalToken": "appr-7f3a9b...",
  "expiresAt": "2026-03-20T10:15:00Z"
}
```

**Client response — approve:**

```json
{
  "type": "cmd", "domain": "signal", "op": "send", "token": "<ATPToken>",
  "body": {
    "signal": "SIGRESUME",
    "target": 57,
    "payload": {
      "approvalToken": "appr-7f3a9b...",
      "decision": "approved",
      "note": "Looks good, send it"
    }
  }
}
```

**Client response — deny:**

```json
{
  "type": "cmd", "domain": "signal", "op": "send", "token": "<ATPToken>",
  "body": {
    "signal": "SIGRESUME",
    "target": 57,
    "payload": {
      "approvalToken": "appr-7f3a9b...",
      "decision": "denied",
      "reason": "Don't send to that address"
    }
  }
}
```

**Kernel on approval:**

- Atomically consumes approvalToken
- Sends `SIGRESUME { decision: "approved" }` to agent
- Agent executes the tool call
- Pushes `hil.resolved` event

**Kernel on denial:**

- Atomically consumes approvalToken
- Sends `SIGRESUME { decision: "denied", reason: "..." }` to agent
- Agent injects the denial as a tool result into LLM context — LLM decides next step
- Pushes `hil.resolved` event

**No CapabilityToken change** — the tool was already granted; this was only an execution gate.

### Scenario 2 — Capability Upgrade

The agent requests a tool it does NOT currently hold in its CapabilityToken. The kernel determines it could be granted (user ACL permits it) but policy requires HIL before modifying the token.

**Flow:**

```
Agent sends ResourceRequest to kernel:
  resource: tool
  name: send_email
  reason: "Need to notify user when analysis complete"
  urgency: low

Kernel evaluates:
  ├── send_email in user's permitted tools? → yes
  ├── blocked by crew deniedTools? → no
  ├── policy: cap_upgrade_requires_hil? → yes
  │
  ├── Mints ApprovalToken bound to (pid:57, tool:send_email)
  ├── Writes /proc/57/hil-queue/hil-002.yaml
  ├── Sends SIGPAUSE to agent
  └── Pushes hil.request (type: capability_upgrade)
```

**`hil.request` event body:**

```json
{
  "hilId": "hil-002",
  "pid": 57,
  "agentName": "researcher",
  "type": "capability_upgrade",
  "tool": "send_email",
  "reason": "Agent requested: Need to notify user when analysis complete",
  "prompt": "Agent is requesting access to send_email. Grant?",
  "grantOptions": [
    { "scope": "once",    "label": "Allow this one call only" },
    { "scope": "session", "label": "Allow for rest of this session" },
    { "scope": "deny",    "label": "Deny" }
  ],
  "approvalToken": "appr-cap-8a2b...",
  "expiresAt": "2026-03-20T10:15:00Z"
}
```

**Client response:**

```json
{
  "type": "cmd", "domain": "signal", "op": "send", "token": "<ATPToken>",
  "body": {
    "signal": "SIGRESUME",
    "target": 57,
    "payload": {
      "approvalToken": "appr-cap-8a2b...",
      "decision": "approved",
      "scope": "session"
    }
  }
}
```

**Kernel on approval:**

- Atomically consumes approvalToken
- Issues a **new CapabilityToken** to PID 57:
  - All previous grants preserved
  - `send_email` added to `spec.tools.granted`
  - If `scope: once` → adds `constraints.tool_call_budget: { send_email: 1 }`
  - Same expiry as current token
  - New HMAC signature
- Delivers new token to agent via `SIGRESUME` payload:
  
  ```json
  { "decision": "approved", "new_capability_token": "<new signed token>" }
  ```
- Agent RuntimeExecutor replaces its held CapabilityToken
- Agent retries the ResourceRequest — now auto-approved
- Pushes `hil.resolved` event

**Kernel on denial:**

- Sends `SIGRESUME { decision: "denied" }` — no token change
- Agent receives denial in ResourceResponse and decides next step

### Scenario 3 — Agent-Initiated Escalation

The agent itself sends `SIGESCALATE` to the kernel, proactively asking for human
guidance before proceeding. No tool approval needed — this is open-ended judgment.

**Flow:**

```
Agent sends SIGESCALATE to kernel:
  {
    reason: "Found PII in dataset. Unsure whether to include in report.",
    context: "Researching Q3 financials, found employee salary data...",
    options: [
      { id: "include", label: "Include with redaction" },
      { id: "exclude", label: "Exclude entirely" },
      { id: "ask_owner", label: "Ask the data owner first" }
    ]
  }

Kernel:
  ├── Mints ApprovalToken bound to (pid:57, hil-id)
  ├── Writes /proc/57/hil-queue/hil-003.yaml
  └── Pushes hil.request (type: escalation)
  — agent is already paused; it sent the signal and awaits response
```

**`hil.request` event body:**

```json
{
  "hilId": "hil-003",
  "pid": 57,
  "agentName": "researcher",
  "type": "escalation",
  "reason": "Found PII in dataset. Unsure whether to include in report.",
  "context": "Researching Q3 financials, found employee salary data...",
  "options": [
    { "id": "include", "label": "Include with redaction" },
    { "id": "exclude", "label": "Exclude entirely" },
    { "id": "ask_owner", "label": "Ask the data owner first" }
  ],
  "approvalToken": "appr-esc-9c3d...",
  "expiresAt": "2026-03-20T10:20:00Z"
}
```

**Client response:**

```json
{
  "type": "cmd", "domain": "signal", "op": "send", "token": "<ATPToken>",
  "body": {
    "signal": "SIGRESUME",
    "target": 57,
    "payload": {
      "approvalToken": "appr-esc-9c3d...",
      "decision": "approved",
      "selectedOption": "exclude",
      "guidance": "Yes, exclude salary data. Focus on revenue and margins only."
    }
  }
}
```

**Kernel on response:**

- Atomically consumes approvalToken
- Delivers `SIGRESUME` to agent with full guidance payload
- Agent RuntimeExecutor injects `guidance` as a system-level instruction into the LLM context:
  
  ```
  [Human instruction]: Exclude salary data. Focus on revenue and margins only.
  ```
- Agent resumes from where it paused, now with the guidance in context
- **No CapabilityToken change** — capabilities are unchanged; only instructions updated
- Pushes `hil.resolved` event

### `hil.resolved` Event

Pushed after any HIL is resolved (approved, denied, or timed out):

```json
{
  "type": "event",
  "event": "hil.resolved",
  "body": {
    "hilId": "hil-001",
    "pid": 57,
    "type": "tool_call_approval",
    "outcome": "approved",          // approved | denied | timeout
    "resolvedBy": "alice",          // username of responding client
    "resolvedAt": "2026-03-20T10:06:12Z",
    "note": "Looks good, send it"   // optional human note
  }
}
```

### HIL Queue — /proc/<pid>/hil-queue/

Each pending HIL request is written to `/proc/<pid>/hil-queue/<hil-id>.yaml`. Readable
via ATP `fs.read`. Clients that missed the push event (e.g. reconnecting after the 60s
grace window) should read this path to find any pending requests.

```yaml
apiVersion: avix/v1
kind: HilRequest
metadata:
  hilId: hil-001
  pid: 57
  createdAt: 2026-03-20T10:05:00Z
  expiresAt: 2026-03-20T10:15:00Z

spec:
  type: tool_call_approval      # tool_call_approval | capability_upgrade | escalation
  tool: send_email              # present for tool_call_approval and capability_upgrade
  args:                         # present for tool_call_approval only
    to: team@org.com
    subject: Summary ready
  reason: "Agent wants to send an email"
  urgency: normal               # low | normal | high
  approvalToken: appr-7f3a9b... # consumed on first valid SIGRESUME

status:
  state: pending                # pending | approved | denied | timeout
```

### HIL Security Properties

- **ApprovalToken is single-use.** Atomic compare-and-swap in kernel memory. Replay is impossible.
- **ApprovalToken is context-bound.** `tool_call_approval` tokens are bound to `(pid, tool, args_hash)`. A token for one call cannot approve a different call.
- **ApprovalToken never touches disk.** Kernel restart invalidates all pending HIL requests.
- **HIL requires `signal` domain access.** Minimum role: `user` (own agents). Operators can respond to any agent’s HIL.
- **Timeout auto-denies.** If `hil_timeout` elapses with no response, kernel sends `SIGRESUME { decision: "timeout" }`. Agent treats this as a denial.
- **SIGKILL is always valid.** A client may send `SIGKILL` instead of `SIGRESUME` to terminate the agent rather than approve/deny. No approvalToken required for SIGKILL.

-----

## 8. Access Control Model

### Per-message validation pipeline

```
1. Parse JSON frame
2. Decode and verify HMAC signature          → EAUTH if invalid
3. Check token expiry                        → EEXPIRED if expired
4. Confirm sessionId matches connection      → ESESSION if mismatch
5. Check domain × role matrix               → EPERM if denied
6. Check target resource ownership          → EPERM if user targets other's PID
7. Hard veto /secrets/ and /proc/ writes    → EPERM unconditionally
8. Forward to kernel IPC
```

Steps 2–7 are in-memory only. Target: < 50µs per validation.

## 10. Security Rules

1. **Token per message.** Gateway validates on every `cmd`. Revocation is immediate.
1. **Tokens never touch disk.** Kernel restart invalidates all tokens. Re-authenticate.
1. **`/secrets/` hard veto.** `fs.read`/`fs.write` on `/secrets/` returns `EPERM` unconditionally.
1. **`/proc/` write hard veto.** `fs.write` on `/proc/` returns `EPERM` unconditionally.
1. **Admin port separation.** `sys` (mutating) and `cap` only on port `7701`.
1. **Rate limiting.** Auth: 5 attempts/60s/IP. Commands: 200/s/session. Spawns: bounded by quota.
1. **Replay protection.** Command `id` values tracked per session. Duplicates → `EPARSE`.
1. **TLS only.** Plain `ws://` rejected with `426` except in dev mode.
1. **CORS.** Desktop/CLI: `Origin: localhost` only. Server: `gateway.allowed_origins` in `boot.conf`.
1. **Agent output scoping.** `user`-role clients only receive events for their own agents.
